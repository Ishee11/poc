# ***Контекст***

Система — это **журнал финансовых операций игроков в рамках игровой сессии**.
Она **не моделирует игру**, а фиксирует только ввод/вывод денег через фишки.

---

# ***Тип системы***

> ***Ledger / accounting system с кэшируемой проекцией агрегатов***

---

# ***Модель домена***

## Session

Сущность, задающая:

* жизненный цикл: `active → finished`
* курс: `chipRate` (immutable)
* кэш агрегатов:

  * `totalBuyIn`
  * `totalCashOut`

```text
ВАЖНО:
эти поля — не источник истины,
а производная (cached projection) от операций
```

---

## Operation (источник истины)

Факт изменения состояния.

Типы:

* `BuyIn`
* `CashOut`

Атрибуты:

* `operationID` (идемпотентность)
* `sessionID`
* `playerID`
* `chips`
* `createdAt`

👉 **Operations = единственный источник истины**

---

# ***Проекция (aggregates / cache)***

В `Session` хранятся:

```text
totalBuyIn   = SUM(BuyIn)
totalCashOut = SUM(CashOut)
```

```text
tableChips = totalBuyIn − totalCashOut
```

---

## Свойства агрегатов

* обновляются синхронно при записи операций
* поддерживаются в консистентном состоянии
* могут быть пересчитаны из операций
* используются для быстрых проверок (O(1))

---

# ***Ключевые вычисления***

## Стол

```text
tableChips = totalBuyIn − totalCashOut
```

---

## Результат игрока

```text
profit = totalCashOut(player) − totalBuyIn(player)
```

(считается из операций, не хранится)

---

## Состояние игрока (не хранится)

```text
inGame ⇔ последняя операция игрока = BuyIn
outGame ⇔ CashOut или операций нет
```

👉 определяется только через Operations

---

# ***Инварианты***

## Общие

* операции только при `session.active`
* `chips > 0`
* `operationID` уникален (идемпотентность)

---

## BuyIn

* только базовая валидация (`chips > 0`)
* не зависит от состояния игрока

---

## CashOut

Проверяется:

### 1. Состояние игрока

```text
lastOperation(player) = BuyIn
```

---

### 2. Кратность курсу

```text
chips % chipRate == 0
```

---

### 3. Состояние стола

```text
chips <= tableChips
```

(используется кэш агрегатов Session)

---

## FinishSession

```text
tableChips == 0
```

---

# ***Транзакционная модель (критично)***

Каждая операция выполняется **атомарно**:

```text
BEGIN

1. проверка инвариантов
2. INSERT Operation
3. UPDATE Session.total*

COMMIT
```

---

## Гарантии

* агрегаты всегда согласованы с операциями
* нет состояния, где обновлено только одно
* race condition предотвращаются (через блокировки)

---

# ***Идемпотентность***

* `operationID` уникален
* повторная операция:

```text
не создаёт дубликат
не обновляет агрегаты повторно
```

---

# ***Расхождения (Reconciliation)***

Система допускает:

```text
реальные фишки ≠ расчётные
```

---

## Точка 1 — CashOut

```text
requestedChips > tableChips → ошибка
```

---

## Точка 2 — FinishSession

```text
tableChips > 0 → нельзя завершить
```

---

## Решение

Ручное урегулирование через дополнительные операции:

* `BuyIn`
* `CashOut`

---

# ***Восстановление агрегатов***

Возможно:

```text
rebuild Session.total* из Operations
```

Используется при:

* сбоях
* миграциях
* проверках консистентности

---

# ***Границы ответственности***

## Система отвечает за:

* фиксацию денежных потоков
* расчёт итогов
* контроль “кассы стола”
* контроль возможности выхода игрока

---

## Система НЕ отвечает за:

* баланс игрока в реальном времени
* перемещение фишек между игроками
* соответствие реальному столу
* честность игры

---

# ***Слои (архитектурно)***

## Domain

* `Session` — хранит кэш агрегатов
* `Operation` — источник истины
* инварианты (частично)

---

## UseCase

* оркестрация
* транзакции
* работа с репозиториями
* обеспечение атомарности

---

## Repository / Adapter

* работа с БД
* `SELECT FOR UPDATE`
* `INSERT / UPDATE`

---

# ***Ключевая мысль***

> *Operations — это правда системы*
> *Session — это кэш, который делает эту правду быстрой и удобной*
